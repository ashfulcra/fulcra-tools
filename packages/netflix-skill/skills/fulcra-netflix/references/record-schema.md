# Watched-annotation wire shapes — def, records, IDs, fingerprints

Exact shapes `scripts/netflix_import.py` puts on the wire, and the contracts
downstream consumers can rely on. Nothing here is aspirational — every shape
is what the tested importer actually produces.

## The Watched definition

The importer resolves-or-creates a **DurationAnnotation** definition via
`GET /user/v1alpha1/annotation`, matching on the namespace marker (see below),
and creates it with this exact body if absent:

```json
POST /user/v1alpha1/annotation
{
  "name": "Watched",
  "description": "com.fulcradynamics.annotation.media.watched",
  "annotation_type": "duration",
  "measurement_spec": {
    "measurement_type": "duration",
    "value_type": "duration",
    "unit": null
  },
  "tags": [],
  "spec": null
}
```

**The `tags: []` quirk is load-bearing**: the API returns 422 if `tags` is
omitted, even on duration defs where tags are meaningless. Keep it.

### The namespace-marker contract

The def's `description` field carries the marker
`com.fulcradynamics.annotation.media.watched`, and **the marker — not the
def's UUID — is the identity**:

- The importer finds an existing def by matching `description == marker` AND
  `annotation_type == "duration"`, never by name alone and never by a cached
  UUID (UUIDs go stale across account switches and resets).
- **Consumers must do the same.** Definition UUIDs differ per account, so the
  future group-recommendation agent (or anything else reading the pool) must
  locate each member's Watched data by scanning their annotation defs for this
  marker, then querying `DurationAnnotation/<that-def's-uuid>`. Hard-coding
  any per-account UUID breaks on every account except the one it was copied
  from.

## Example wire record — real importer output

Generated by running the actual `build_record` over row 3 of the test fixture
(`tests/fixtures/slim.csv`, the "Dune: Part Two" row), byte-for-byte except
the def-id placeholder:

```json
{
  "specversion": 1,
  "data": "{\"duration_seconds\": 1, \"external_ids\": {\"content_fingerprint\": \"tv:dune:part-two\", \"occurrence_index\": 0, \"point_in_time\": true, \"raw_date\": \"3/1/24\", \"time_estimated\": true, \"timestamp_confidence\": \"low\"}, \"note\": \"Dune: Part Two\", \"title\": \"Dune\"}",
  "metadata": {
    "data_type": "DurationAnnotation",
    "recorded_at": {
      "start_time": "2024-03-01T12:00:00Z",
      "end_time": "2024-03-01T12:00:01Z"
    },
    "tags": [],
    "source": [
      "com.fulcra.media.netflix.v2.e9dcf4140d0ccedd",
      "com.fulcradynamics.annotation.<your-def-id-placeholder>"
    ],
    "content_type": "application/json"
  }
}
```

Reading it:

- `data` is a JSON *string* (sort-keys-serialized payload), per Fulcra's
  ingest format. `duration_seconds`, `note`, `title`, and `external_ids` live
  inside it.
- **What varies per user**: only `source[1]` — the
  `com.fulcradynamics.annotation.<def-uuid>` element, which embeds that
  account's Watched def UUID. Every other byte is a pure function of the CSV
  content, which is what makes cross-account schema identity (and the pool's
  future group work) possible.
- `source[0]` is the deterministic record ID (det-id) — Fulcra dedups ingest
  on source-id, so this element **is the idempotency key** for re-runs.
- The slim fixture row shows the honest slim-variant caveat in action: this
  is actually the movie *Dune: Part Two*, but the slim export gives no signal
  to distinguish a two-part movie title from a `Show: Episode` pattern, so
  the inferred fingerprint is `tv:dune:part-two` and the title splits at the
  colon. The GDPR variant carries enough context to classify it correctly
  (`movie:dune-part-two`). This is a known, accepted precision limit of the
  slim path — one more reason the GDPR export is the upgrade.
- Records are POSTed to `/ingest/v1/record/batch` as JSONL
  (`content-type: application/x-jsonl`), 500 records per chunk.

## Deterministic record IDs (det-ids)

Both formulas are SHA-256, first 16 hex chars, ported byte-identically from
`fulcra-media` so records dedup perfectly against fulcra-media imports of the
same history.

**Slim variant:**

```
det_id = "com.fulcra.media.netflix.v2." + sha256(f"{date_str}|{raw_title}|{occurrence}")[:16]
```

- `date_str` is the raw `M/D/YY` string from the CSV, `raw_title` the raw
  Title cell.
- `occurrence` is the row's index **among byte-identical (date, title) rows
  in the file**, counted in file order. Rationale: two identical
  `Title,Date` rows in a slim export are a real same-day rewatch, not a
  duplicate — hashing the row alone would silently collapse them. Because
  the index is computed per identical-row *group* (not file position),
  re-importing the same or an overlapping CSV reproduces the same det-ids
  and stays idempotent.

**GDPR (rich) variant:**

```
det_id = "com.fulcra.media.netflix-rich." + sha256(f"{profile}|{start_time_str}|{raw_title}")[:16]
```

- Rich rows carry a real timestamp, so the row is already unique — no
  occurrence index needed (the slim formula's index would degenerate to ~0
  here anyway).

### Version-lockstep warning

The `v2` in the slim prefix pairs with the script's `INGEST_VERSION = 2`.
Fulcra silently drops POSTs whose source-ids match existing records — even
records under soft-deleted defs — so a schema change that alters record
content **without** a prefix bump would be silently swallowed, and a prefix
bump **without** a coordinated fulcra-media change would fork the two tools'
det-id spaces and break their mutual dedup. Therefore: **bump
`INGEST_VERSION` and the det-id prefix ONLY together, and ONLY in lockstep
with `fulcra-media`'s Netflix importer** (`packages/media-helpers/
fulcra_media/importers/netflix.py`). Never bump casually.

## Content fingerprints

`data.external_ids.content_fingerprint` is a cross-source content identifier
(scheme shared with `fulcra_common.cross_source_fingerprint`), so twin-dedup
tooling can match this record against the same title imported from Trakt,
Apple TV, etc. — even though det-ids differ across tools. Shapes:

| Pattern in the Netflix title | Fingerprint |
|---|---|
| `Movie Name` | `movie:movie-name` |
| `Show: Season 1: Episode 5` | `tv:show:s01e05` |
| `BEEF: Season 2: Episode Title` | `tv:beef:s02:episode-title` |
| `Anthology: 2021: Episode Title` | `tv:anthology:s2021:episode-title` |
| `Show: Limited Series: Episode 1` | `tv:show:limited-series:e01` |
| `Show: Episode Title` (2-part) | `tv:show:episode-title` |
| `Dune: Part Two` (2-part, known-movie context — GDPR only) | `movie:dune-part-two` |

Slugs are lowercase, alphanumeric-and-hyphen, runs collapsed. The fingerprint
deliberately lives in `data.external_ids` (not the `source` array) to match
where fulcra-media's importer puts it — same shape, same place, regardless of
which tool wrote the record.

## Dedup summary

- **Within this importer**: source-id (det-id) dedup, server-side. Re-runs
  are no-ops; the batch endpoint returns no dedup feedback, which is why the
  envelope's `skipped_existing` is always `null` and `posted` counts POST
  attempts, not novel records.
- **Across tools**: `content_fingerprint` matching — det-ids intentionally
  differ between tools, fingerprints intentionally agree.
